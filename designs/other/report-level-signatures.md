## 1. Overview

### 1.1 Purpose

Display electronic signature information on printed and PDF patient laboratory reports generated by OpenELIS Global. When results have been electronically signed during result entry and validation workflows, the report must show the complete signature chain in the Signature / Validation area of the report, replacing the current blank handwritten signature field.

### 1.2 Scope

This specification covers:

- Display of electronic signature data on the patient results report (the "Results of Analysis" report)
- Formatting of signature blocks to meet §11.50 manifestation requirements on printed output
- Behavior when some results on a report are signed and others are not
- Behavior when e-signatures are disabled at the site level
- Integration with the existing report generation pipeline

### 1.3 Out of Scope

- Digitally signed PDF files (cryptographic PDF signatures / PKI)
- Signature capture on screen during report printing
- Signatures on non-patient reports (QC reports, aggregate/statistical reports, audit reports)
- E-signature ceremony at time of printing (printing is a read-only operation)

### 1.4 Dependencies

- Electronic Signature for 21 CFR Part 11 Compliance (FRS v1.0) — the `electronic_signature` table and related data model must be implemented first
- Existing OpenELIS report generation infrastructure (Jasper Reports / report templates)

### 1.5 Regulatory Reference

- **§11.50 — Signature Manifestations:** "Signed electronic records shall contain information associated with the signing that clearly indicates: (a) The printed name of the signer; (b) The date and time when the signature was executed; (c) The meaning associated with the signature (such as review, approval, responsibility, or authorship)."
- This requirement applies to **any rendering** of a signed electronic record, including printed and PDF output.

## 2. Design Goals

| Goal | Description |
|------|-------------|
| **§11.50 Compliance** | Every rendering of a signed result — on-screen, printed, or PDF — must display the signer's printed name, timestamp, and signature meaning |
| **Backward Compatibility** | When e-signatures are disabled, the report retains its current layout with the blank Signature / Validation box for handwritten signatures |
| **Clear Attribution** | Auditors and clinicians can immediately identify who authored and who validated each result on the report |
| **Minimal Template Changes** | Modify the existing report template's Signature / Validation section rather than restructuring the entire report layout |
| **Mixed-State Support** | Reports may contain a mix of signed, unsigned, in-progress, and canceled results — the signature block must handle this clearly |

## 4. Signature Display Requirements

### 4.1 Per-Result Signatures vs. Report-Level Signatures

Electronic signatures in OpenELIS are captured **per result** (each test result has its own signature records). The report, however, may contain many results across multiple test sections. The report signature block must aggregate this information clearly.

**Approach: Report-level summary with per-result attribution available**

The Signature / Validation section of the report displays a **summary signature block** listing each unique signer and their role. Individual result rows in the results table get a visual indicator showing their signature status.

### 4.2 Mandatory Display Elements (§11.50)

For every electronic signature associated with results on the report, the following must appear:

| Element | Format | Source |
|---------|--------|--------|
| Printed name of signer | Full name as text (not a handwritten or image signature) | `electronic_signature.signer_name_printed` |
| Date and time of signature | `DD/MM/YYYY HH:MM UTC` (matching OpenELIS date convention) | `electronic_signature.signed_at` |
| Meaning of signature | "Authored", "Validated and Released", or "Rejected" | `electronic_signature.signature_meaning` |

### 4.3 Additional Display Elements

| Element | Purpose |
|---------|---------|
| Rejection reason | Required for rejected results — displays in the signature block and/or inline with the result row |
| "Electronically signed" indicator | Distinguishes e-signatures from handwritten — important for auditors |

```
┌─────────────────────────────────────────────────────────────────┐
```

```
│                                                                  │
```

```
│   Dr. Sarah Chen — 09/02/2026 10:20 UTC                        │
│                                                                  │
│ ─ Electronically Signed ─                                        │
```

Rejection reasons and affected tests remain visible in the results table rows (e.g., "Reject reason 17/01/2025 03:39 : The sample received is jaundiced...") and are not repeated in the signature block to avoid redundancy.

### 5.5 Layout — No Electronic Signatures (Results Not Yet Validated)

When the report is printed before any validation has occurred (e.g., a tech printing for manual review):

```
┌─────────────────────────────────────────────────────────────────┐
│ Signature / Validation                                           │
│                                                                  │
│ Results on this report have not been electronically signed.      │
│                                                                  │
│ Authorized signature: ____________________________________       │
│                                                                  │
│ Printed name: ____________________________________________       │
│                                                                  │
│ Date: ____________________________________________________       │
└─────────────────────────────────────────────────────────────────┘
```

This preserves the handwritten signature workflow for results that haven't been through e-signature validation yet.

### 5.6 Signature Block Placement

The signature block (along with General Comments) is **not** fixed to a dedicated second page. It flows naturally after the last result row and renders wherever space allows:

- If results fit on page 1 with room remaining, the signature block appears at the bottom of page 1
- If results span multiple pages, the signature block appears after the last result row on the final page
- The current behavior of forcing a separate page 2 for the signature area is removed — this reduces page count for shorter reports

The signature block always appears **after all result sections** and **before the Legend / Date of Report footer**.

### 5.7 Result-Row Signature Status Indicator

In addition to the summary signature block, each result row in the results table gets a **Status** column indicator:

| Current Status Values | New E-Sig Indicator | Meaning |
|----------------------|---------------------|---------|
| (blank — result present) | No change when e-sig disabled | Result has been entered |
| "In progress" | No change | Result pending |
| "Canceled" | No change | Result canceled |
| (blank — result present, e-sig enabled) | `✓` appended to status or shown in Alert column | Result has been validated & released with e-signature |

**Note:** The primary §11.50 compliance requirement is met by the signature block. The row-level indicator is a usability enhancement — it lets readers quickly identify which specific results on a partial report have been formally validated.

### 5.8 Report Header Layout

The report header displays between the `headerLeftImage` and `headerRightImage` logos, centered. It uses three existing Printed Report Configuration fields to provide site-specific identification on every page:

```
┌──────────────────────────────────────────────────────────────────────┐
│  [headerLeftImage]     Results of Analysis          [headerRightImage]│
│                      Results Final Report                             │
│                   Vet + Clinica Veterinária          ← SiteName       │
│                  Lab Director: Dr. Gil Izabel        ← labDirectorTitle│
│                          Vet mais                    ← additional      │
│                                                        site info      │
└──────────────────────────────────────────────────────────────────────┘
```

#### 5.8.1 Header Fields (Existing Configuration)

These fields already exist in **Administration → Printed Report Configuration** and require no new configuration entries:

| Config Field | Usage in Header | Example |
|-------------|----------------|---------|
| `SiteName` | Lab/site name, bold, centered below report title | Vet + Clinica Veterinária |
| `labDirectorTitle` | Title prefix for lab director line | Dr. |
| `labDirectorName` | First/given name of lab director | Gil |
| `lab director` | Surname of lab director | Izabel |
| `additional site info` | Free-text line below the director line | Vet mais |

The lab director line is composed as: `"Lab Director: {labDirectorTitle} {labDirectorName} {lab director}"`. If all three lab director fields are empty, the line is omitted. If `additional site info` is empty, that line is omitted. If `SiteName` is empty, only the report title ("Results of Analysis" / "Results Final Report" / "Results Partial Report") appears.

#### 5.8.2 Header on Continuation Pages

When the report spans multiple pages, each continuation page repeats a **compact header** that includes:

- `headerLeftImage` and `headerRightImage` logos
- Report title (compact, smaller font)
- `SiteName`
- Lab director line
- `additional site info`
- Abbreviated patient identification: Patient code, National ID, Lab Number, Prescriber, Referring site + Ward

This ensures every printed page is identifiable if pages are separated.

### 5.9 Page Layout and Print Specifications

#### 5.9.1 Target Paper Sizes

The report template is optimized for **A4** as the primary target, which is the more constrained of the two common paper sizes. Reports also render correctly on US Letter.

| Paper Size | Dimensions | Usage |
|-----------|-----------|-------|
| **A4** (primary) | 210 × 297 mm (8.27 × 11.69 in) | Standard in most countries, including all OpenELIS deployments outside the US |
| **US Letter** | 215.9 × 279.4 mm (8.5 × 11 in) | Standard in the US; wider but shorter than A4 |

#### 5.9.2 Margins and Content Area

| Parameter | Value | Notes |
|-----------|-------|-------|
| Top margin | 15 mm | Accommodates the report header on every page |
| Bottom margin | 15 mm | Accommodates the Legend / Date of Report footer |
| Left margin | 15 mm | |
| Right margin | 15 mm | |
| Content width | 180 mm | 210mm (A4 width) − 30mm (left + right margins) |
| Content height per page | ~267 mm | 297mm (A4 height) − 30mm (top + bottom margins) |

#### 5.9.3 Page Break Rules

The Jasper Report template must enforce the following page break rules to ensure the printed report is clean and professional:

| Rule | Implementation | Rationale |
|------|---------------|-----------|
| **Signature block + footer never split** | The General Comments / Electronic Signatures box and the Legend / Date of Report footer are grouped together with `page-break-inside: avoid` (Jasper: `isRemoveLineWhenBlank` + frame grouping) | The signature block is the most critical compliance element — splitting it across pages could create ambiguity about which signatures apply to which results |
| **Result rows never split mid-row** | Each result table row (including any associated reject reason or correction note immediately below it) uses `page-break-inside: avoid` | A result value separated from its test name across a page boundary is unreadable |
| **Department section headers keep with next** | Department headings (e.g., "Hematology") use `page-break-after: avoid` to stay with at least the first result row of that section | A section title alone at the bottom of a page with all its results on the next page looks broken |
| **Table headers repeat on continuation pages** | The result table column headers (Test, Spec, Result, Status, Alert, Reference value, Unit) repeat at the top of each continuation page | Readers must be able to identify columns on every page without flipping back to page 1 |
| **Report header repeats on continuation pages** | The compact header (§5.8.2) renders at the top of each continuation page | Every page must be identifiable if pages become separated |

#### 5.9.4 Overflow Behavior

When content exceeds a single page:

1. Results continue on the next page, preceded by the compact header and abbreviated patient info
2. Result table column headers repeat
3. The signature block and footer render on the final page after the last result row
4. If the signature block does not fit in the remaining space on the final results page, it moves entirely to a new page (it never splits)

#### 5.9.5 Font Sizing for Print

Font sizes are optimized for print legibility at A4 scale:

| Element | Font Size | Notes |
|---------|----------|-------|
| Report title ("Results of Analysis") | 16–18pt | Prominent, centered |
| Report subtitle (Final/Partial) | 10–12pt | Below title |
| Header lines (SiteName, director, additional info) | 9–10pt | Below subtitle |
| Patient info table | 9pt | Compact but readable at arm's length |
| Result table headers | 9pt bold | |
| Result table values | 9pt | |
| Reject reason / correction notes | 8pt italic | Subordinate to the result row |
| Signature block entries | 8–9pt | |
| Legend / Date of Report | 8pt | |
| Compliance footer | 7–8pt | Smallest element; regulatory reference only |

## 6. Behavior by Report State

### 6.1 Decision Matrix

| E-Sig Feature | Report Has Validated Results | Report Has Unsigned Results | Signature Block Display |
|--------------|-----------------------------|-----------------------------|------------------------|
| Disabled | N/A | N/A | **Current layout** — blank "Signature / Validation" box with Date field (no change) |
| Enabled | Yes — all results signed | No | **Full e-sig block** — all signers listed, "Electronically Signed" footer |
| Enabled | Yes — some results signed | Yes — some in progress/unsigned | **Partial e-sig block** — signers listed + partial report note + blank lines for manual signature of remaining |
| Enabled | No | Yes — all unsigned | **Unsigned block** — "not been electronically signed" message + blank handwritten signature lines |
| Enabled | Yes — some rejected | Mixed | **E-sig block with rejection section** — includes rejection reason and affected tests |

### 6.2 Report Title

When e-signatures are enabled, the report title reflects the completion and signing state:

| Condition | Report Title |
|-----------|-------------|
| All non-canceled results have been validated with e-signatures | **Results Final Report** |
| Some results are still pending, in progress, or unsigned | **Results Partial Report** (unchanged from current) |
| E-signatures disabled | **Results Partial Report** or current title behavior (unchanged) |

The "Results Final Report" title, combined with the "Electronically Signed" footer in the signature block, conveys an official, authoritative document. No additional suffix (e.g., "— Electronically Signed") is needed in the title since the signature block itself provides that context.

### 6.3 Reprint Behavior

- Reprinted reports always reflect the **current** signature state of results at the time of printing
- If a result was validated after the first print, the reprint shows the updated signature block
- Each print includes "Date of Report" timestamp at the bottom to distinguish versions
- The report does not store signature state at time of first print — it always queries current state

## 7. Configuration

### 7.1 Site-Level Settings

These settings are part of the broader e-signature configuration (see Electronic Signature FRS §13):

| Setting | Type | Default | Description |
|---------|------|---------|-------------|
| `esig.enabled` | Boolean | `false` | Master toggle — when false, report uses current blank signature layout |
| `esig.report.show_compliance_footer` | Boolean | `true` | Show "21 CFR Part 11 Compliant" text in signature block |
| `esig.report.show_authored_signatures` | Boolean | `true` | Include "Authored by" signatures on report (some labs may only want validation signatures visible to clinicians) |

### 7.2 Lab Director Signature Image

Laboratories may have a standing policy where the lab director's signature (as an image) appears on all final reports as the authorizing official. This is already partially configured via the existing Printed Report Configuration fields (`labDirectorSignature`, `labDirectorName`, `labDirectorTitle`, `lab director`). The e-signature feature adds a toggle to control whether this image appears on reports.

#### 7.2.1 Existing Configuration (No Changes Needed)

These fields already exist in **Administration → Printed Report Configuration**:

| Existing Config Field | Type | Description |
|----------------------|------|-------------|
| `labDirectorSignature` | Image (PNG/JPG) | Uploaded image of the lab director's signature |
| `labDirectorName` | String | Lab director's first/given name |
| `labDirectorTitle` | String | Title prefix (e.g., "Dr.") |
| `lab director` | String | Lab director's surname |

#### 7.2.2 New Configuration

One new setting controls whether the lab director image signature appears on the report:

| Setting | Type | Default | Description |
|---------|------|---------|-------------|
| `esig.report.show_lab_director_signature` | Boolean | `true` | Enable/disable the lab director image signature on reports |

#### 7.2.3 Admin UI

The existing admin screen at **Administration → Printed Report Configuration** already provides:

- An image upload field for the signature image (`labDirectorSignature`)
- Text fields for the lab director's name and title (`labDirectorName`, `labDirectorTitle`, `lab director`)

The e-signature toggle (`esig.report.show_lab_director_signature`) can be added to the existing Printed Report Configuration screen or to a dedicated e-signature configuration section.

#### 7.2.5 Behavior Matrix

| Lab Director Image Enabled | E-Sig Enabled | Result |
|---------------------------|---------------|--------|
| Yes | Yes | Image signature at top of block, e-sig entries below |

| No | Yes | E-sig entries only, no image |

### 7.3 Configuration Rationale

- **show_authored_signatures:** Some laboratory workflows consider the "Authored" signature an internal QA step. Clinicians receiving the report may only need to see the validation signature. When `false`, "Authored by" entries are omitted from the printed report signature block (they remain in the database and on-screen manifestation for audit purposes).
- **show_compliance_footer:** Labs not subject to FDA regulation but using e-signatures for internal quality purposes may not want the "21 CFR Part 11" reference on reports sent to external clinicians.

## 8. Report Generation Integration

### 8.1 Data Source

The report template queries the `electronic_signature` table to populate the signature block:

```sql
SELECT DISTINCT
    es.signer_name_printed,
    es.signature_meaning,
    MIN(es.signed_at) AS earliest_signature,
    MAX(es.signed_at) AS latest_signature
FROM electronic_signature es
WHERE es.record_type = 'RESULT'
  AND es.record_id IN (:resultIdsOnReport)
GROUP BY es.signer_name_printed, es.signature_meaning
ORDER BY
    CASE es.signature_meaning
        WHEN 'AUTHORED' THEN 1
        WHEN 'VALIDATED_AND_RELEASED' THEN 2
        WHEN 'REJECTED' THEN 3
    END,
    earliest_signature ASC;
```

### 8.2 Template Changes

**Affected template:** The patient results report template (Jasper Report / `.jrxml` file)

**Changes required:**

1. **Conditional rendering** of the signature block based on `esig.enabled` site setting
2. **New sub-report or detail band** for the electronic signature block, replacing the static "Signature / Validation" box
3. **Parameter passing** of aggregated signature data to the signature sub-report
4. **Partial report note** logic — check if any result on the report has status "In progress" or is missing a validation signature

### 8.3 Signature Block Sub-Report Data

The sub-report receives a collection of signature summary records:

| Field | Type | Description |
|-------|------|-------------|
| `signerNamePrinted` | String | Full name of signer |
| `signatureMeaning` | Enum | AUTHORED, VALIDATED_AND_RELEASED, REJECTED |
| `signedAt` | Timestamp | When signature was executed |

Additionally, the sub-report receives:

| Field | Type | Description |
|-------|------|-------------|
| `hasUnsignedResults` | Boolean | True if any result on report lacks a validation signature |
| `allResultsSigned` | Boolean | True if every non-canceled, non-in-progress result has both authored and validation signatures |
| `showAuthoredSignatures` | Boolean | From site config `esig.report.show_authored_signatures` |
| `showComplianceFooter` | Boolean | From site config `esig.report.show_compliance_footer` |
| `reportLocale` | Locale | From system site language setting — drives message key resolution and date/time formatting |
| `labDirectorSignatureImage` | Image/Bytes | From existing config `labDirectorSignature` (null if not configured) |
| `labDirectorName` | String | Composed from existing config `labDirectorTitle` + `labDirectorName` + `lab director` |
| `showLabDirectorSignature` | Boolean | From site config `esig.report.show_lab_director_signature` |

## 9. Internationalization (i18n)

All display strings introduced by this feature must be localized using the existing OpenELIS Global message properties framework (`messages_XX.properties`). No user-visible string may be hardcoded in the report template or application code.

### 9.1 Message Keys — Signature Block

| Message Key | English (Default) | Context |
|------------|-------------------|---------|
| `report.esig.header` | Electronic Signatures | Signature box header when e-signatures are present |

### 9.6 Locale and Date/Time Format Resolution

The report template resolves the locale from the site-configured language setting (the same mechanism used by all other OpenELIS reports). The locale determines:

- Which `messages_XX.properties` file is used for all display strings
- Date/time formatting for signature timestamps — inherited from the system date format configured in **Administration → General Configurations → Site Information**

Signature timestamps use the same date format as all other dates on the report (e.g., `DD/MM/YYYY HH:MM` for most locales, `MM/DD/YYYY HH:MM` for US English). There is no separate date format override for e-signature timestamps — they inherit from the existing system-wide report date format setting. This ensures consistency across the entire report.

If a translation is missing for a given locale, the system falls back to the English default.

### 9.7 Non-Localized Content

The following content is **not** localized because it is user-entered or system-generated data, not UI labels:

| Content | Reason |
|---------|--------|
| Signer's printed name | Stored as entered during e-signature ceremony |
| Signature timestamps | Formatted per system locale date format setting |
| `SiteName`, `additional site info` | Admin-entered site-specific content |
| `labDirectorTitle`, `labDirectorName`, `lab director` | Admin-entered personal name |
| Rejection reason text | Entered by the rejecting user |
| Patient data, result values, test names | Existing data from the OpenELIS data model |

## 10. User Interface Changes

### 10.1 Print Report Screen

No changes to the print/export workflow. Printing remains a read-only operation — it does not trigger an e-signature ceremony. The report is generated with whatever signature data currently exists for the results.

When viewing the report on screen before printing, the same signature block rendering applies. The on-screen preview matches the printed output exactly.

This feature does not introduce new screens. It modifies the output of the existing patient report template.
